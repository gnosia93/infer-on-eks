
영상은 모델에 "동영상 파일 그대로" 들어가지 않는다. 핵심은 프레임으로 쪼개서 각 프레임을 이미지 토큰으로 변환한 후, 텍스트 토큰과 합쳐서 LLM에 넣는 흐름이다.

#### [전체 파이프라인] ####
```
동영상 파일 (mp4 등)
   ↓  ① 디코딩 & 프레임 추출 (샘플링)
여러 장의 프레임 이미지 [f1, f2, ..., fN]
   ↓  ② 전처리 (resize, normalize)
   ↓  ③ 비전 인코더 (ViT 등) → 패치 단위 임베딩
프레임별 비주얼 토큰
   ↓  ④ 프로젝터(MLP)로 LLM 임베딩 공간에 정렬
   ↓  ⑤ 텍스트 토큰과 하나의 시퀀스로 결합
LLM (1B) → 출력
```

### 각 단계 설명 ###
#### ① 프레임 추출 (샘플링) #### 
영상의 모든 프레임(30fps면 1분에 1,800장)을 다 넣지 않고, 일정 개수만 샘플링한다. 이 N(프레임 수)이 토큰 수와 VRAM을 직접 결정한다.

* 균등 샘플링: 전체에서 N장 균등하게 뽑기 (예: 8, 16, 32, 64장)
* FPS 기반: "초당 1프레임"처럼 고정 비율로 추출


#### ② 전처리 각 프레임을 모델이 요구하는 해상도로 resize하고 정규화 ###
해상도가 높을수록 프레임당 토큰이 증가한다 (예: 336×336 → 576토큰).

#### ③ 비전 인코더 각 프레임을 패치(예: 14×14)로 나눠 ViT가 임베딩으로 변환 ####
프레임 1장 → 수백 개의 비주얼 토큰.

#### ④ 프로젝터 정렬 비전 임베딩을 LLM이 이해하는 임베딩 차원으로 매핑 ####
일부 모델은 여기서 토큰 압축(token merging/pooling)을 해서 프레임당 토큰을 줄인다. (메모리 절약의 핵심 기법).

#### ⑤ 텍스트와 결합 최종 입력 시퀀스는 대략 다음과 같은 형태가 된다. ####
```
[시스템 프롬프트] <video> f1_tokens f2_tokens ... fN_tokens </video> [질문 텍스트]
```

### 실제 API로는 어떻게 보내나 ###

#### 1. 서버가 영상을 직접 디코딩 ####
```
# 예: Qwen2-VL 스타일
messages = [{
    "role": "user",
    "content": [
        {"type": "video", "video": "file:///path/video.mp4", "fps": 1.0},
        {"type": "text", "text": "이 영상에서 무슨 일이 일어나?"}
    ]
}]
```

#### 2. 클라이언트가 프레임을 미리 뽑아서 이미지 리스트로 전달 ####
```
content = [
    {"type": "image", "image": frame1},
    {"type": "image", "image": frame2},
    # ... N장
    {"type": "text", "text": "이 프레임들을 순서대로 보고 설명해줘"}
]
```

### 메모리 최적화 ###

* 총 비주얼 토큰 = 프레임 수(N) × 프레임당 토큰 수

이게 KV 캐시 크기를 결정하므로, VRAM을 줄이는 레버는

* 프레임 수 줄이기 (64 → 16장)
* 해상도 낮추기 (프레임당 토큰 감소)
* 토큰 압축/풀링 지원 모델 선택 (Qwen2-VL, InternVL 등)

## GPU 메모리 산정 ##

1B 멀티모달 모델 인퍼런스의 GPU 메모리는 모델 가중치 보다 입력이 만들어내는 토큰 수(특히 영상)가 좌우한다.

### 메모리 구성 요소 ###

#### 1. 모델 가중치 (1B 기준) ####

* FP16/BF16 ->약 2GB
* INT8 ->	약 1GB
* INT4 -> 약 0.5GB

여기서 주의할 점은 "1B"가 LLM 부분만인지, 비전 인코더(ViT 등)까지 포함인지 확인이 필요한데, 보통 비전 인코더가 0.3~0.4B 추가로 붙게된다.

#### 2. KV 캐시 + 액티베이션 ####

멀티모달은 이미지/영상을 토큰으로 변환하는데, 이 토큰 수가 메모리를 결정한다.

* 이미지 1장: 보통 256~1,024 토큰 → 부담 적음
* 영상: 프레임 수 × 프레임당 토큰. 예를 들어 프레임당 256토큰 × 32프레임 = 8,000토큰 이상으로 쉽게 커지게 된다.

1B급(레이어 ~24, hidden ~2048) 기준 FP16 KV 캐시는 토큰당 대략 200KB 내외라서:

* 8,000토큰 → 약 1.5GB
* 32,000토큰(긴 영상) → 약 6GB+

#### 3. 실무 권장 사항 ####

* 이미지 전용: 12~16GB면 충분히 여유 (RTX 4060 Ti 16GB, L4 24GB 등)
* 영상 처리: 24GB 권장 (L4 24GB, RTX 4090 24GB, A10G 24GB). 긴 영상이나 다중 동시 요청이 있으면 더 필요
* 여러 요청 동시 처리(배치/서빙): 40~48GB (A6000, A100 40GB) 고려

모델명 Qwen2-VL, InternVL, LLaVA, SmolVLM 등 아키텍처마다 토큰 처리 방식이 다르다.
* 영상 길이 / 프레임 샘플링 수 (예: 8프레임 vs 64프레임 — 메모리 차이가 큼)
* 동시 처리 요청 수(concurrency)와 배치 크기
* 정밀도 (FP16인지 양자화 적용인지)
* 서빙 프레임워크 (vLLM, TGI, transformers 등) — vLLM은 PagedAttention으로 KV 캐시 효율이 좋다.

가장 큰 변수는 영상 길이(프레임 수) 이다.

## 아키텍처별 토큰 처리 방식 ##

핵심 차이는 이미지/프레임 한 장을 몇 개의 토큰으로 바꾸느냐, 그리고 그 개수가 고정이냐 가변이냐로 나뉘며, 이게 곧 KV 캐시와 액티베이션 메모리를 결정하게 된다.
모든 VLM은 ViT로 패치 임베딩 → 어떤 방식으로든 토큰 압축 → LLM 입력이라는 구조가 같지만, ② 압축 방식과 ① 해상도 처리 방식에서 갈리게 된다.

### 토큰 수를 줄이는 3가지 대표 기법 ###

| 기법 | 동작 | 대표 모델 |
| :--- | :--- | :--- |
| **Pixel Shuffle**<br>*(space-to-depth)* | 인접 패치를 채널 방향으로 접어 토큰 수를 1/4, 1/16로 압축 | InternVL, SmolVLM |
| **Patch Merging** | 2×2 인접 패치를 하나로 병합 (4배 감소) | Qwen2-VL |
| **Pooling / 그대로** | 압축 거의 없이 ViT 패치 그대로 사용 | LLaVA-1.5 |


### 모델별 토큰 처리 방식 ###
#### LLaVA (특히 1.5) ####

* 고정 토큰: 336×336 입력 → 24×24 = 576토큰/이미지 (압축 거의 없음)
* 해상도 가변성 없음(1.5 기준). LLaVA-NeXT는 타일링 추가
* 영상(LLaVA-NeXT-Video): 프레임마다 576토큰, 다만 pooling으로 줄이기도 함
* 특징: 단순하지만 프레임당 토큰이 많아 영상에서 메모리 부담이 큼

#### Qwen2-VL ####

* Naive Dynamic Resolution: 해상도에 비례해 토큰 수가 유동적으로 변함
* 2×2 patch merging으로 4배 압축
* min_pixels/max_pixels 설정으로 토큰 상한을 직접 제어 가능 (메모리 튜닝에 매우 유리)
* 영상: 3D 위치 인코딩(시간축 포함) + 프레임 병합. 긴 영상도 비교적 효율적
* 특징: 토큰 예산을 세밀하게 조절할 수 있어 서빙 시 예측/제어가 쉬움

#### InternVL ####

* Dynamic High-Resolution 타일링: 이미지를 종횡비에 맞춰 여러 448×448 타일로 분할(예: 최대 12타일 + 썸네일 1장)
* 각 타일은 pixel shuffle로 압축 → 타일당 약 256토큰
* 그래서 총 토큰 = 타일 수 × 256. 고해상도 문서/디테일에 강하지만 타일이 많아지면 토큰이 급증
* 특징: 디테일 인식은 좋지만 타일 수를 안 잡으면 메모리가 튐

#### SmolVLM (SmolVLM2가 영상 지원) ####

* 극단적 압축이 설계 목표. aggressive pixel shuffle로 프레임/이미지당 토큰을 매우 낮게 (프레임당 수십~81토큰 수준)
* 저메모리 환경(엣지, 소형 GPU)을 노린 모델
* 특징: 같은 프레임 수여도 토큰이 훨씬 적어 VRAM 최소 (대신 세밀한 디테일은 손해)

### 메모리에 주는 영향 (동일 조건 비교) ###
"16프레임 영상"을 넣는다고 가정하면, 프레임당 토큰 수 차이가 그대로 총 토큰 → KV 캐시로 이어진다.

| 모델 | 프레임당 토큰 (근사) | 16프레임 총 비주얼 토큰 | 상대적 VRAM |
| :--- | :--- | :--- | :--- |
| **SmolVLM** | 64 / 81 | 1,000 ~ 1,300 | 최소 |
| **Qwen2-VL**<br>*(max_pixels 제한 시)* | ~수백 (조절 가능) | 조절 가능 | 낮음 ~ 중간 |
| **InternVL**<br>*(타일 1~2장/프레임)* | 256 / 512 | 4,000 ~ 8,000 | 중간 |
| **LLaVA-1.5 계열** | ~576 | ~9,000 | 높음 |


앞서 계산한 대로 1B급에서 KV 캐시는 대략 **토큰당 ~200KB(FP16)**이므로:


* SmolVLM 1,300토큰 → 약 0.25GB
* LLaVA 9,000토큰 → 약 1.7GB
* (긴 영상으로 토큰이 3만 넘어가면 6GB+ 로 급증)

### 결론 / 선택 가이드 ###
* VRAM을 최소화하고 싶다 → SmolVLM2 (극단 압축, 엣지/소형 GPU 적합)
* 토큰 예산을 정밀하게 제어하며 영상 효율 좋게 → Qwen2-VL (max_pixels, 프레임 수로 튜닝)
* 문서/고해상도 디테일이 중요 → InternVL (단, 타일 수 제한 필수)
* 단순 구현/생태계 호환 → LLaVA 계열 (대신 영상에선 토큰이 많아 메모리 비쌈)

즉 "1B라서 다 비슷하겠지"가 아니라, 어떤 압축 방식을 쓰느냐에 따라 같은 영상이어도 VRAM이 수 배 차이 나게 된다.

#### L4 24GB 또는 RTX 4090 24GB 한 장이면 1B급 + 영상 인퍼런스의 가장 무난한 선택이고, vLLM으로 서빙하면 KV 캐시 관리까지 효율적이다. 나중에 동시 처리량이 부족하면 그때 L40S/A100으로 스케일업하면 된다. ####
