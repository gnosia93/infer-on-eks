❯ vllm 모델을 다운로드 받아서 어디다가 저장하는거야?

⏺ 기본 저장 위치 — Hugging Face 캐시

  vLLM은 모델 다운로드를 **Hugging Face transformers/huggingface_hub**에 위임합니다. 그래서 저장 위치도 HF 캐시 규칙을
  따라요. 기본값:

  ~/.cache/huggingface/hub/

  그 안에 모델별로 폴더가 생깁니다:

  ~/.cache/huggingface/hub/
  └── models--Qwen--Qwen2.5-32B-Instruct-FP8/
      ├── blobs/          ← 실제 파일 내용 (해시 이름)
      ├── snapshots/      ← 특정 리비전의 파일들 (blobs로의 심볼릭 링크)
      │   └── <commit_hash>/
      │       ├── config.json
      │       ├── model-00001-of-000xx.safetensors
      │       └── ...
      └── refs/           ← 브랜치/리비전 포인터

  - blobs에 진짜 데이터가 있고, snapshots는 링크로 연결 — 같은 파일을 리비전끼리 공유해 중복 저장을 피하는 구조예요.

  g6e.2xlarge에서 주의할 점 ⚠️

  이게 AWS 인스턴스에서 실질적으로 중요한 문제입니다.

  - 홈 디렉토리(~)는 보통 루트 EBS 볼륨에 있는데, 이게 작으면(예: 30~50GB) 32B 모델 다운로드 중 디스크가 꽉 참.
  - 32B FP8 = ~33GB, FP16 = ~65GB. 루트 볼륨 여유 없으면 다운로드 실패.

  해결: 큰 볼륨에 캐시 위치를 옮기세요.

  # 별도 EBS 볼륨을 /data에 마운트했다고 가정
  export HF_HOME=/data/hf
  vllm serve Qwen/Qwen2.5-32B-Instruct-FP8 ...

  저장 위치를 바꾸는 환경변수

  ┌────────────────────┬───────────────────────────────────────┬──────────────────────────────┐
  │      환경변수      │                 역할                  │           우선순위           │
  ├────────────────────┼───────────────────────────────────────┼──────────────────────────────┤
  │ HF_HOME            │ HF 전체 홈 ($HF_HOME/hub에 모델 저장) │ 이거 하나만 바꿔도 됨 (권장) │
  ├────────────────────┼───────────────────────────────────────┼──────────────────────────────┤
  │ HF_HUB_CACHE       │ 모델 캐시 경로 직접 지정              │ HF_HOME보다 구체적           │
  ├────────────────────┼───────────────────────────────────────┼──────────────────────────────┤
  │ TRANSFORMERS_CACHE │ (구버전) 캐시 경로                    │ 레거시, 위 둘로 대체         │
  └────────────────────┴───────────────────────────────────────┴──────────────────────────────┘

  보통 HF_HOME 하나만 큰 볼륨으로 지정하면 됩니다.

  다운로드 없이 로컬 경로에서 바로 로딩

  이미 받아둔 모델이나 직접 만든 체크포인트가 있으면, 모델 이름 대신 경로를 주면 됩니다:

  vllm serve /data/models/Qwen2.5-32B-Instruct-FP8

  → 이땐 다운로드 안 하고 그 폴더에서 바로 읽습니다.

  확인/관리 팁

  # 캐시에 뭐가 있나 보기 (huggingface_hub 설치 시)
  huggingface-cli scan-cache

  # 캐시 지우기
  huggingface-cli delete-cache

  # 지금 캐시 경로 확인
  echo $HF_HOME
  du -sh ~/.cache/huggingface/hub/*

  참고 — gated 모델 (Llama 등)

  Llama 같은 모델은 HF 계정 승인 + 토큰이 필요합니다:

  export HF_TOKEN=hf_xxxxx
  # 또는 huggingface-cli login

  Qwen 계열은 대부분 공개라 토큰 없이 받아집니다.

  ---
  한 줄 요약: vLLM은 HF 캐시(~/.cache/huggingface/hub/)에 models--조직--모델명/ 구조로 저장합니다. g6e.2xlarge에서는 루트
  볼륨이 작아 32B가 안 들어갈 수 있으니 HF_HOME을 큰 EBS 볼륨으로 지정하세요. 이미 받은 모델은 경로를 직접 줘서 다운로드
  없이 로딩할 수 있습니다.

  ⚠️ 환경변수 우선순위나 캐시 구조 세부는 huggingface_hub 버전에 따라 조금씩 달라질 수 있으니, 실제로 huggingface-cli
  scan-cache로 확인하는 게 확실합니다.

✻ Crunched for 24s

❯ 실제 운영 환경에서 예를들어서 g6e. 여러대로 여러대의 vllm 으로 서빙한다고 할때. 이걸 개별 ebs 로 저장하는거야? 아님. 공용
  스토리지를 중간에 끼우는거야? 예르들어서 efs.. 같은거.

⏺ 좋은 운영 질문이에요. 결론부터 말하면 "공용 스토리지에서 받아서 로컬로 복사(또는 캐시)하는 하이브리드"가 정석이고, EFS를
  모델 로딩 경로로 직접 쓰는 건 대체로 피합니다. 이유가 명확해요.

  핵심 원칙: "저장은 공용, 로딩은 로컬"

  모델 파일에는 두 가지 상충하는 요구가 있습니다:

  - 배포·관리: 여러 노드가 같은 모델을 공유해야 함 → 공용 스토리지가 편함
  - 로딩 속도: 33~65GB를 GPU에 빠르게 올려야 함 → 로컬 디스크가 압도적으로 빠름

  그래서 실무는 둘을 결합합니다: 공용에 원본을 두고, 각 노드는 로컬로 가져와서 로딩.

  옵션별 비교

  ┌──────────────────────────┬───────────┬───────────┬─────────────────────┬─────────────┐
  │           방식           │ 로딩 속도 │   비용    │      운영 편의      │    평가     │
  ├──────────────────────────┼───────────┼───────────┼─────────────────────┼─────────────┤
  │ 개별 EBS에 각자 다운로드 │ 빠름 ✅   │ 중복 저장 │ 노드마다 재다운로드 │ 소규모엔 OK │
  ├──────────────────────────┼───────────┼───────────┼─────────────────────┼─────────────┤
  │ EFS를 직접 로딩 경로로   │ 느림 ❌   │ 비쌈      │ 공유 쉬움           │ ⚠️ 비권장   │
  ├──────────────────────────┼───────────┼───────────┼─────────────────────┼─────────────┤
  │ S3 → 로컬 복사 후 로딩   │ 빠름 ✅   │ 저렴      │ 좋음                │ ✅ 정석     │
  ├──────────────────────────┼───────────┼───────────┼─────────────────────┼─────────────┤
  │ EFS/FSx → 로컬 캐시      │ 빠름 ✅   │ 중간      │ 좋음                │ ✅ 좋음     │
  └──────────────────────────┴───────────┴───────────┴─────────────────────┴─────────────┘

  왜 EFS를 "직접 로딩 경로"로 쓰면 안 되나

  vllm serve /mnt/efs/model 처럼 EFS를 그대로 가리키면:

  - 처리량 병목: EFS는 네트워크 파일시스템이라 대용량 순차 읽기가 로컬 NVMe보다 훨씬 느림. 33GB 로딩이 수 분씩 걸릴 수 있음.
  - 동시 로딩 폭발: 오토스케일로 여러 vLLM 노드가 동시에 같은 EFS에서 모델을 읽으면 대역폭 경쟁 → 전부 느려짐. 스케일아웃
  순간이 가장 취약.
  - 비용: EFS는 GB·처리량 과금이라 대용량 반복 읽기가 비쌈.
  - 작은 파일 IOPS엔 강하지만, 대용량 순차엔 약함 — 모델 로딩은 정확히 후자.

  → 공유는 편하지만 로딩 성능·비용에서 손해라 로딩 경로로는 부적합.

  실무에서 가장 흔한 패턴들

  패턴 1: S3를 진실의 원천(source of truth)으로 (가장 일반적)

  S3 버킷 (모델 원본 보관)
     │  노드 부팅/시작 시 다운로드
     ▼
  각 노드 로컬 EBS/NVMe  →  vLLM이 여기서 로딩

  - 모델을 S3에 한 번 올려두고, 각 노드가 시작할 때 aws s3 cp 또는 s5cmd(병렬 다운로드, 훨씬 빠름)로 로컬에 내림.
  - vLLM은 로컬 경로를 가리킴 → 빠른 로딩.
  - HF에서 매번 받는 것보다 빠르고(같은 리전 S3), 버전 통제도 됨.
  - g6e는 로컬 NVMe 인스턴스 스토리지가 없으니(EBS만) 로컬 EBS를 충분히 크게 잡아둬야 함.

  패턴 2: EFS/FSx는 "스테이징"용, 로딩은 로컬

  - EFS/FSx에 원본을 두되, 직접 로딩하지 않고 노드 시작 시 로컬로 cp 후 로딩.
  - FSx for Lustre는 EFS보다 대용량 처리량이 훨씬 좋아서, 큰 모델·많은 노드면 FSx를 선호하기도 함(특히 S3 연동 FSx).

  패턴 3: 컨테이너 이미지에 모델 굽기 (소수 모델 고정 시)

  - Docker 이미지 안에 모델을 포함시켜 배포.
  - 장점: 별도 다운로드 없음, 버전이 이미지에 고정.
  - 단점: 이미지가 수십 GB로 비대해짐 → 모델 자주 안 바뀔 때만.

  g6e 다중 노드에 대한 구체 권장

  당신 상황(g6e 여러 대, 각자 단일 GPU vLLM, 오토스케일 가능성)이라면:

  1. 모델 원본은 S3에 (같은 리전, 사전 양자화 FP8 체크포인트로)
  2. 노드 시작 시 s5cmd로 로컬 EBS에 병렬 다운로드 → vLLM은 로컬 경로 로딩
  3. 로컬 EBS는 모델 크기 + 여유(예: 100GB+ gp3)로 넉넉히
  4. 다운로드를 부팅 스크립트/initContainer(K8s)에 넣어 자동화

  이렇게 하면:
  - 스케일아웃 시 각 노드가 S3에서 병렬로 받아 EFS 같은 공유 병목 없음
  - 로딩은 로컬이라 빠름
  - 모델 교체는 S3 객체만 갱신하면 됨

  캐시 공유 관련 디테일

  - HF 캐시(HF_HOME)를 EFS에 공유해서 "한 번만 다운로드하면 모두가 씀"을 노리는 사람도 있는데, 위의 동시 로딩 병목 문제로
  로딩 성능이 나빠집니다. 다운로드 절약 ≠ 로딩 성능. 권장 안 함.
  - 정말 공유 캐시를 쓰려면 FSx for Lustre처럼 고처리량 FS를 쓰거나, "공유는 원본 보관만, 로딩은 로컬 복사"로 분리하세요.

  ---
  한 줄 요약: EFS를 vLLM 로딩 경로로 직접 끼우는 건 대용량 순차 읽기가 느리고 동시 스케일아웃 때 병목·비용 문제가 커서
  비권장입니다. 정석은 S3(또는 FSx)에 모델 원본을 두고, 각 노드가 시작 시 로컬 EBS로 복사(s5cmd 병렬)한 뒤 로컬 경로에서
  로딩하는 하이브리드예요. 공용 스토리지는 "보관·배포"용, 로컬 디스크는 "로딩"용으로 역할을 나누는 게 핵심입니다.

  ⚠️ EFS vs FSx 처리량 특성, g6e의 EBS 대역폭 한도 등 구체 수치는 워크로드와 볼륨 타입에 따라 다르니 실제 벤치로 확인하세요.
  필요하면 s5cmd 다운로드 스크립트나 K8s initContainer 예시를 짜드릴게요.

✻ Sautéed for 41s

❯  노드 시작 시 s5cmd로 로컬 EBS에 병렬 다운로드. 이 부분이 구체적으로 어떻게 동작하는지 잘모르겟어.

⏺ 좋아요, 이 부분을 처음부터 구체적으로 풀어볼게요. "노드가 시작될 때 자동으로 S3에서 모델을 로컬 디스크로 내려받고, 그 다음
  vLLM을 띄운다"는 흐름입니다.

  먼저 s5cmd가 뭔지 — 왜 aws s3 cp가 아니라 이걸 쓰나

  aws s3 cp도 되지만, s5cmd는 병렬 처리에 특화된 훨씬 빠른 S3 클라이언트입니다.

  - 32B 모델은 .safetensors 파일이 여러 조각(model-00001-of-00014.safetensors ...)으로 나뉘어 있음
  - aws s3 cp는 기본적으로 다소 보수적으로 받음
  - s5cmd는 여러 파일 + 각 파일의 여러 부분을 동시에 내려받아서, 같은 네트워크 대역폭을 훨씬 꽉 채움 → 실측으로 몇 배 빠른
  경우 많음

  33GB를 받을 때 이 차이가 "10분 vs 3분" 수준으로 벌어질 수 있어요.

  전체 그림

  [노드 부팅]
     │
     ▼
  [시작 스크립트 자동 실행]
     │  1. s5cmd로 S3 → 로컬 EBS 병렬 다운로드
     │  2. 다운로드 완료 확인
     │  3. vllm serve (로컬 경로) 실행
     ▼
  [서버 준비 완료 → 트래픽 받음]

  핵심은 "다운로드"와 "서버 실행"을 순서대로 엮은 시작 스크립트를 노드가 뜰 때 자동 실행시키는 거예요. 그 "자동 실행" 방법이
  환경(그냥 EC2냐, K8s냐)에 따라 다릅니다.

  기본 다운로드 명령

  # S3의 모델 폴더 전체를 로컬로 병렬 다운로드
  s5cmd cp 's3://my-models-bucket/Qwen2.5-32B-Instruct-FP8/*' \
          /data/models/Qwen2.5-32B-Instruct-FP8/

  - * 와일드카드 → 그 경로 아래 모든 파일을 동시에 받음
  - --concurrency 로 병렬도 조절 가능 (파일 내부 분할 다운로드 수)

  s5cmd --numworkers 32 cp 's3://.../model/*' /data/models/model/

  - --numworkers: 동시에 처리할 작업 수 (파일 많을수록 효과)

  시나리오 A: 순수 EC2 (Auto Scaling Group)

  EC2는 User Data 스크립트를 노드 최초 부팅 때 자동 실행합니다. 여기에 다운로드+실행을 넣습니다.

  #!/bin/bash
  set -euo pipefail

  MODEL_S3="s3://my-models-bucket/Qwen2.5-32B-Instruct-FP8"
  MODEL_LOCAL="/data/models/Qwen2.5-32B-Instruct-FP8"

  # 1. 로컬 디렉토리 준비 (EBS가 /data에 마운트되어 있다고 가정)
  mkdir -p "$MODEL_LOCAL"

  # 2. 병렬 다운로드
  s5cmd --numworkers 32 cp "${MODEL_S3}/*" "$MODEL_LOCAL/"

  # 3. vLLM 실행 (로컬 경로 로딩 — 다운로드 안 함)
  vllm serve "$MODEL_LOCAL" \
    --quantization fp8 \
    --gpu-memory-utilization 0.92 \
    --max-model-len 8192

  - 이 스크립트가 ASG로 새 노드가 뜰 때마다 자동 실행 → 각 노드가 알아서 S3에서 받아 로컬에 준비하고 서버를 띄움.
  - IAM: 이 EC2의 인스턴스 역할에 해당 S3 버킷 읽기 권한이 있어야 s5cmd가 인증 없이 접근 가능.

  시나리오 B: Kubernetes (EKS) — initContainer 패턴

  K8s면 "다운로드"를 initContainer로, "서빙"을 메인 컨테이너로 분리하는 게 정석입니다. initContainer는 메인 컨테이너보다
  먼저, 완료될 때까지 실행돼요.

  apiVersion: apps/v1
  kind: Deployment
  spec:
    template:
      spec:
        volumes:
          - name: model-volume
            emptyDir: {}          # 또는 로컬 EBS 기반 volume
        initContainers:
          - name: model-downloader
            image: peakcom/s5cmd:latest
            command:
              - /bin/sh
              - -c
              - |
                s5cmd --numworkers 32 cp \
                  's3://my-models-bucket/Qwen2.5-32B-Instruct-FP8/*' \
                  /models/Qwen2.5-32B-Instruct-FP8/
            volumeMounts:
              - name: model-volume
                mountPath: /models
        containers:
          - name: vllm
            image: vllm/vllm-openai:latest
            args:
              - "--model=/models/Qwen2.5-32B-Instruct-FP8"
              - "--quantization=fp8"
              - "--gpu-memory-utilization=0.92"
            volumeMounts:
              - name: model-volume
                mountPath: /models        # initContainer가 받은 걸 그대로 봄

  동작 순서:
  1. initContainer가 먼저 실행 → s5cmd로 공유 volume(/models)에 다운로드
  2. 다운로드 성공하면 종료
  3. 그제서야 vLLM 컨테이너가 시작 → 같은 volume의 로컬 경로에서 모델 로딩

  두 컨테이너가 volume을 공유하니, initContainer가 받은 파일을 vLLM이 그대로 읽습니다.

  왜 이 구조가 좋은가

  - 역할 분리: 다운로드 실패하면 서버가 아예 안 뜸 → "모델 없는 채로 트래픽 받는" 사고 방지 (initContainer가 성공해야
  다음으로 넘어가므로)
  - 스케일아웃 독립성: 새 노드가 각자 S3에서 병렬로 받음 → EFS 같은 공유 병목 없음. S3는 사실상 무제한 병렬 읽기를 감당.
  - 로딩 빠름: 받고 나면 로컬 디스크에서 로딩

  현실적으로 신경 쓸 것들

  ① 콜드 스타트 시간
  새 노드가 뜰 때마다 33GB 다운로드 + 로딩 → 수 분 걸림. 그래서:
  - 오토스케일 반응이 느릴 수 있으니 스케일업을 미리 트리거(예측 스케일링)
  - 또는 미리 데워진(pre-warmed) 노드 풀 유지

  ② 중복 다운로드 캐싱
  같은 노드가 재시작할 때 매번 다시 받으면 낭비.
  - 다운로드 전에 이미 로컬에 있는지 체크하고 스킵:
  if [ ! -f "$MODEL_LOCAL/config.json" ]; then
    s5cmd --numworkers 32 cp "${MODEL_S3}/*" "$MODEL_LOCAL/"
  fi
  - 단, ASG의 새 인스턴스는 EBS가 비어있으니 결국 받아야 함. AMI에 모델을 미리 구워두는 방법도 있음(트레이드오프).

  ③ 무결성 확인
  다운로드가 중간에 끊기면 손상된 파일로 로딩 실패. s5cmd는 크기 비교로 재시도하지만, 중요하면 파일 개수·크기 검증을
  스크립트에 넣으세요.

  ④ EBS 대역폭
  g6e의 EBS 처리량 한도가 다운로드 속도 상한이 될 수 있음. 병렬도를 높여도 EBS 쓰기 속도에서 막히면 그 이상 안 빨라짐. gp3
  볼륨의 throughput을 넉넉히 프로비저닝하세요.

  정리 흐름

  S3 (모델 원본, 리전 내)
     │  노드 시작 시 s5cmd 병렬 다운로드 (User Data or initContainer)
     ▼
  로컬 EBS (/data/models/...)   ← 이미 있으면 스킵
     │  vllm serve 로컬 경로
     ▼
  vLLM 서버 준비 완료

  내려받고, 완료되면 그 로컬 경로로 vllm serve를 실행"하는 구조입니다. s5cmd가 여러 파일을 동시에 받아 대역폭을 꽉 채우므로
