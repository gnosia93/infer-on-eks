 1. 먼저 알아야 할 배경 지식

  vLLM을 제대로 이해하려면 LLM 인퍼런스의 두 단계를 먼저 알아야 합니다.

  - Prefill (prompt 처리): 입력 전체를 한 번에 처리 → compute-bound
  - Decode (토큰 생성): 한 토큰씩 autoregressive 생성 → memory-bound, KV 캐시가 계속 커짐

  이 구조 때문에 KV 캐시 관리가 핵심 문제라는 걸 이해하는 게 출발점입니다.

  2. vLLM의 핵심 아이디어 3가지

  ┌───────────────────┬──────────────────────────────────────────┬────────────────────────────────────┐
  │       개념        │                   무엇                   │              왜 중요               │
  ├───────────────────┼──────────────────────────────────────────┼────────────────────────────────────┤
  │ PagedAttention    │ KV 캐시를 OS 가상메모리처럼 페이지 블록  │ 메모리 단편화 제거 → 훨씬 많은     │
  │                   │ 단위로 관리                              │ 동시 요청 처리                     │
  ├───────────────────┼──────────────────────────────────────────┼────────────────────────────────────┤
  │ Continuous        │ 요청을 토큰 단위로 배치에 넣고 뺌        │ GPU 유휴시간 최소화, 처리량 대폭   │
  │ Batching          │                                          │ 상승                               │
  ├───────────────────┼──────────────────────────────────────────┼────────────────────────────────────┤
  │ Prefix Caching    │ 공통 프롬프트 KV 재사용                  │ system prompt 반복 시 효율         │
  └───────────────────┴──────────────────────────────────────────┴────────────────────────────────────┘

  이 중 PagedAttention 논문("Efficient Memory Management for LLM Serving", SOSP 2023)이 가장 중요합니다.
  이거 하나는 꼭 읽으세요.

  3. 실습 로드맵

  1단계 — 그냥 돌려보기
  pip install vllm
  from vllm import LLM, SamplingParams

  llm = LLM(model="meta-llama/Llama-3.2-1B-Instruct")  # 작은 모델로 시작
  out = llm.generate(["안녕하세요, 오늘 날씨는"], SamplingParams(temperature=0.8, max_tokens=64))
  print(out[0].outputs[0].text)

  2단계 — OpenAI 호환 서버 띄우기
  vllm serve meta-llama/Llama-3.2-1B-Instruct
  # curl로 localhost:8000/v1/chat/completions 호출

  3단계 — 옵션 만져보며 원리 체감
  - --gpu-memory-utilization, --max-num-seqs, --max-model-len
  - --quantization awq, --enable-prefix-caching
  - --tensor-parallel-size (멀티 GPU)

  4단계 — 코드 읽기
  - vllm/engine/llm_engine.py (스케줄러 흐름)
  - vllm/core/scheduler.py (continuous batching 실제 구현)
  - vllm/attention/ (PagedAttention 커널)

❯ 서빙 최적화, 서빙하기 위주.

⏺ 서빙 위주라면 개념보다 **"어떤 상황에서 어떤 옵션을 켜느냐"**가 핵심입니다. 실무 관점으로
  정리해드릴게요.

  1. 먼저 목표를 정하세요 — 처리량 vs 지연시간

  모든 튜닝은 이 트레이드오프에서 갈립니다.

  ┌────────────────────────────────┬──────────────────────┬───────────────────────────────────────────┐
  │              목표              │         성향         │                 대표 옵션                 │
  ├────────────────────────────────┼──────────────────────┼───────────────────────────────────────────┤
  │ Throughput (배치 처리, 대량    │ 배치 크게, GPU 꽉    │ --max-num-seqs ↑,                         │
  │ 요청)                          │ 채움                 │ --gpu-memory-utilization ↑                │
  ├────────────────────────────────┼──────────────────────┼───────────────────────────────────────────┤
  │ Latency (챗봇, 실시간)         │ 배치 작게, 응답      │ speculative decoding, --max-num-seqs      │
  │                                │ 빠르게               │ 적당히                                    │
  └────────────────────────────────┴──────────────────────┴───────────────────────────────────────────┘

  2. 핵심 서빙 커맨드

  vllm serve meta-llama/Llama-3.1-8B-Instruct \
    --gpu-memory-utilization 0.90 \
    --max-model-len 8192 \
    --max-num-seqs 256 \
    --enable-prefix-caching \
    --tensor-parallel-size 1

  3. 성능에 직접 영향 주는 옵션 (우선순위 순)

  ① --gpu-memory-utilization (0.9 기본)
  KV 캐시로 쓸 GPU 메모리 비율. 높일수록 더 많은 동시 요청 처리. OOM 안 나는 선에서 최대한 올리기
  (0.90~0.95).

  ② --max-model-len
  지원할 최대 컨텍스트 길이. 불필요하게 크게 잡으면 KV 캐시 예약 때문에 동시 처리량이 줄어듭니다. 실제
  필요한 만큼만.

  ③ --max-num-seqs
  동시에 배치에 들어갈 최대 시퀀스 수. 처리량 튜닝의 핵심 노브.

  ④ --enable-prefix-caching
  system prompt나 공통 프리픽스가 반복되면 무조건 켜세요. RAG·챗봇에서 효과 큼.

  ⑤ --quantization (awq / gptq / fp8)
  메모리 부족하거나 더 큰 모델 올리고 싶을 때. FP8은 H100 이상에서 품질 손실 적음.

  ⑥ --tensor-parallel-size
  모델이 GPU 하나에 안 들어갈 때 여러 GPU로 분할. (예: 70B 모델 → 4)

  ⑦ Speculative Decoding (지연시간 줄이기)
  --speculative-model "[ngram]" --num-speculative-tokens 5
  # 또는 draft 모델 지정

  4. 반드시 해야 할 것 — 벤치마킹

  옵션 추측하지 말고 측정하세요. vLLM이 벤치 스크립트를 제공합니다.

  # 서버 띄운 상태에서
  python benchmarks/benchmark_serving.py \
    --backend vllm \
    --model meta-llama/Llama-3.1-8B-Instruct \
    --dataset-name sharegpt \
    --num-prompts 1000 \
    --request-rate 10

  봐야 할 지표:
  - TTFT (Time To First Token) — 첫 토큰까지, 체감 반응성
  - TPOT / ITL (Time Per Output Token) — 토큰 생성 속도
  - Throughput (req/s, token/s)
  - P50 / P99 latency — 꼬리 지연

  5. 서빙 실무에서 자주 만나는 이슈

  - OOM: --gpu-memory-utilization 낮추거나 --max-model-len 줄이기
  - 처리량이 안 나옴: --max-num-seqs 올리고, prefix caching 켜고, request-rate 높여서 배치가 실제로 차는지
  확인
  - TTFT가 김 (긴 프롬프트): chunked prefill(--enable-chunked-prefill)로 prefill과 decode 겹치기
  - 꼬리 지연(P99) 튐: 배치 크기 제한, chunked prefill로 완화

  6. 프로덕션 배포

  - Docker: 공식 vllm/vllm-openai 이미지 사용
  - Kubernetes: 오토스케일링 + readiness probe
  - 모니터링: vLLM이 /metrics로 Prometheus 지표 노출 (큐 길이, KV 캐시 사용률, throughput 등)
  - 로드밸런싱: 여러 replica 앞에 LB, prefix-aware routing 하면 캐시 적중률↑

✻ Brewed for 30s

❯ g6e.2xlarge 쓸꺼야.

⏺ 좋습니다. g6e.2xlarge 사양부터 맞춰서 구체적으로 정리해드릴게요.

  g6e.2xlarge 스펙 (이게 모든 결정의 기준)

  ┌───────────────┬─────────────────┬────────────────────────────────────────────────────┐
  │     항목      │       값        │                        의미                        │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────┤
  │ GPU           │ NVIDIA L40S 1장 │ 단일 GPU → TP=1 (tensor parallel 불필요)           │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────┤
  │ VRAM          │ 48GB GDDR6      │ 모델 + KV 캐시가 여기 다 들어가야 함               │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────┤
  │ 아키텍처      │ Ada Lovelace    │ FP8 지원 ✅ (품질 손실 적게 양자화 가능)           │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────┤
  │ 메모리 대역폭 │ ~864 GB/s       │ H100(3.35TB/s)보다 낮음 → decode가 상대적으로 느림 │
  ├───────────────┼─────────────────┼────────────────────────────────────────────────────┤
  │ vCPU / RAM    │ 8 / 64GB        │ CPU 오프로딩 여유는 적음                           │
  └───────────────┴─────────────────┴────────────────────────────────────────────────────┘

  핵심 포인트 2가지:
  1. 48GB에 뭘 올릴 수 있나가 모델 선택을 결정합니다.
  2. NVLink 없고 대역폭이 H100 대비 낮아서, 대역폭이 병목인 decode 최적화(FP8, speculative decoding)가
  특히 효과적입니다.

  48GB에 들어가는 모델 (대략)

  ┌────────────────┬──────────┬───────────┬────────────────────────────┐
  │      모델      │   FP16   │  FP8/AWQ  │            추천            │
  ├────────────────┼──────────┼───────────┼────────────────────────────┤
  │ 8B (Llama 3.1) │ ~16GB    │ ~8GB      │ ✅ 여유롭게, KV 캐시 넉넉  │
  ├────────────────┼──────────┼───────────┼────────────────────────────┤
  │ 14B (Qwen2.5)  │ ~28GB    │ ~14GB     │ ✅ 좋음                    │
  ├────────────────┼──────────┼───────────┼────────────────────────────┤
  │ 32B (Qwen2.5)  │ ~64GB ❌ │ ~18GB     │ ✅ FP8/AWQ로만             │
  ├────────────────┼──────────┼───────────┼────────────────────────────┤
  │ 70B            │ ❌       │ ~40GB빠듯 │ ⚠️ KV 캐시 거의 없음, 비추 │
  └────────────────┴──────────┴───────────┴────────────────────────────┘

  FP16 기준 모델 크기 = 파라미터 × 2GB. 나머지가 KV 캐시로 쓰여서 동시 처리량을 결정합니다. 48GB 중 모델이
  많이 먹을수록 동시 요청 수가 줄어요.

  추천 시작 커맨드

  14B FP8 (밸런스 좋은 출발점):
  vllm serve Qwen/Qwen2.5-14B-Instruct \
    --quantization fp8 \
    --gpu-memory-utilization 0.92 \
    --max-model-len 8192 \
    --max-num-seqs 128 \
    --enable-prefix-caching \
    --enable-chunked-prefill

  8B FP16 (가장 안전, 처리량 높음):
  vllm serve meta-llama/Llama-3.1-8B-Instruct \
    --gpu-memory-utilization 0.92 \
    --max-model-len 8192 \
    --max-num-seqs 256 \
    --enable-prefix-caching

  L40S 특화 튜닝 포인트

  - FP8을 적극 활용: Ada 아키텍처라 하드웨어 FP8 지원. 메모리도 아끼고 대역폭 병목인 decode도 빨라집니다.
  L40S에서 가성비 최고 옵션.
  - --tensor-parallel-size는 건들지 마세요: GPU 1장이라 TP=1(기본값). 여러 장 있는 것처럼 설정하면
  에러납니다.
  - Speculative decoding이 잘 먹힘: 대역폭 병목 환경이라 draft 모델로 decode 지연 줄이는 효과가 큼.
  지연시간 민감하면 시도.
  - KV 캐시 양자화: 긴 컨텍스트 필요하면 --kv-cache-dtype fp8로 KV 캐시도 줄여서 동시 처리량 확보.

  주의할 점

  - --gpu-memory-utilization을 너무 높이면(0.95+) CUDA graph·활성값 버퍼 여유가 없어 OOM. 0.90~0.92에서
  시작하세요.
  - 8 vCPU밖에 없어서 전처리/토크나이징이 병목될 수 있음. 요청 동시성 아주 높게 잡을 땐 CPU도 모니터링.
  - 단일 GPU라 이 인스턴스로 70B는 사실상 무리. 필요하면 g6e.12xlarge(L40S 4장) 이상으로 가야 합니다.

  ---
  1. 챗봇 — 저지연 우선 (Latency)

  목표: TTFT 짧게, 토큰이 빠르게 흘러나오게. 배치를 작게 유지해 각 요청이 GPU를 빨리 차지하도록.

  vllm serve meta-llama/Llama-3.1-8B-Instruct \
    --gpu-memory-utilization 0.90 \
    --max-model-len 8192 \
    --max-num-seqs 64 \
    --enable-prefix-caching \
    --enable-chunked-prefill \
    --speculative-config '{"method":"ngram","num_speculative_tokens":5,"prompt_lookup_max":4}'

  논리:
  - --max-num-seqs 64 — 낮게. 배치 커지면 개별 응답이 느려짐. 저지연은 처리량을 일부 포기.
  - Speculative decoding — L40S는 대역폭 병목이라 decode 가속 효과 큼. ngram은 draft 모델 없이 프롬프트
  반복 패턴으로 예측(챗봇/코드에서 잘 먹힘).
  - 작은 모델(8B) 선택 — 응답 속도가 곧 UX. FP16로 품질 유지.
  - Chunked prefill — 긴 프롬프트 들어와도 다른 유저의 decode가 안 막힘 → TTFT 안정.

  볼 지표: TTFT, ITL(토큰 간 지연), P99.

  ---
  2. 배치 처리 — 고처리량 우선 (Throughput)

  목표: 초당 최대 토큰. 지연은 신경 안 씀. GPU를 항상 꽉 채우는 게 전부.

  vllm serve Qwen/Qwen2.5-14B-Instruct \
    --quantization fp8 \
    --gpu-memory-utilization 0.94 \
    --max-model-len 4096 \
    --max-num-seqs 512 \
    --enable-chunked-prefill \
    --max-num-batched-tokens 8192

  또는 오프라인 배치라면 서버 없이:
  from vllm import LLM, SamplingParams
  llm = LLM(model="Qwen/Qwen2.5-14B-Instruct", quantization="fp8",
            gpu_memory_utilization=0.94, max_num_seqs=512)
  outputs = llm.generate(prompts, SamplingParams(max_tokens=512))  # 수천 개 한 번에

  논리:
  - --max-num-seqs 512 — 크게. 동시 시퀀스 많을수록 GPU utilization↑.
  - --gpu-memory-utilization 0.94 — 최대한. KV 캐시 공간을 확보해 배치를 키움.
  - FP8 양자화 — 모델을 줄여 남는 메모리를 전부 KV 캐시로 → 더 많은 동시 요청.
  - --max-model-len 4096 — 실제 필요한 만큼만. 짧을수록 요청당 KV 캐시가 작아 배치가 커짐.
  - Speculative decoding 끔 — 대량 배치에선 이미 GPU가 꽉 차서 오히려 오버헤드.
  - 오프라인이면 LLM.generate() — 서버 오버헤드 없이 최고 처리량.

  볼 지표: total throughput(req/s, token/s), GPU utilization.

  ---
  3. RAG — 긴 프롬프트 + 반복 프리픽스

  목표: 검색된 긴 문맥을 효율적으로. 공통 프리픽스 재사용이 핵심.

  vllm serve Qwen/Qwen2.5-14B-Instruct \
    --quantization fp8 \
    --gpu-memory-utilization 0.92 \
    --max-model-len 32768 \
    --max-num-seqs 128 \
    --enable-prefix-caching \
    --enable-chunked-prefill \
    --kv-cache-dtype fp8

  논리:
  - --enable-prefix-caching 필수 — RAG는 system prompt·지시문이 매 요청 반복됨. 프리픽스 KV 재사용으로
  prefill 비용 대폭 절감. 같은 문서를 여러 질문에 쓰면 그 문서 KV도 캐시됨.
  - --max-model-len 32768 크게 — 검색 문맥이 길어서 컨텍스트 넉넉히. 단, 클수록 KV 캐시 예약↑ →
  --max-num-seqs는 128로 타협.
  - --kv-cache-dtype fp8 — 긴 컨텍스트는 KV 캐시가 메모리를 크게 먹음. FP8로 절반 줄여 동시 처리량 확보.
  L40S가 FP8 지원하니 활용.
  - Chunked prefill 중요 — RAG는 prefill이 무거움(긴 입력). 잘게 나눠 처리해 TTFT와 다른 요청 응답성 유지.
  - FP8 모델 — 32K 컨텍스트에선 메모리가 빡빡하므로 모델부터 줄여 여유 확보.

  볼 지표: TTFT(긴 프롬프트라 중요), prefix cache hit rate(/metrics에서 확인).

⏺ AWQ = Activation-aware Weight Quantization

  논문 제목 그대로 **"활성값을 고려한 가중치 양자화"**입니다. 모델 가중치를 **INT4(4비트)**로 압축하는
  양자화 기법이에요.

  핵심 아이디어

  일반적인 양자화의 문제: 가중치를 무작정 4비트로 줄이면 정확도가 크게 떨어집니다. 특히 일부 가중치는 모델
  성능에 훨씬 중요한데, 이걸 뭉개면 품질이 망가져요.

  AWQ의 통찰:

  ▎ "어떤 가중치가 중요한지는 가중치 자체가 아니라, 그 가중치에 곱해지는 활성값(activation)의 크기를 보면
  ▎ 알 수 있다."

  - 활성값이 큰 채널에 연결된 가중치 = 중요한(salient) 가중치 (전체의 ~1%)
  - 이 중요한 가중치는 양자화 오차를 줄여주고, 나머지는 과감히 4비트로 압축
  - 실제로는 중요한 채널을 scaling으로 보호한 뒤 양자화 (별도 데이터 타입 안 쓰고 하드웨어 친화적)

  → 결과: 4비트인데도 품질 손실이 매우 적음.

  왜 쓰는가 (효과)

  ┌─────────────┬───────────────────┬───────────────────────┐
  │    항목     │       FP16        │      AWQ (INT4)       │
  ├─────────────┼───────────────────┼───────────────────────┤
  │ 가중치 크기 │ 파라미터 × 2 byte │ 파라미터 × ~0.5 byte  │
  ├─────────────┼───────────────────┼───────────────────────┤
  │ 32B 모델    │ ~65GB             │ ~18GB                 │
  ├─────────────┼───────────────────┼───────────────────────┤
  │ 품질        │ 기준              │ 거의 유지 (소폭 저하) │
  └─────────────┴───────────────────┴───────────────────────┘

  메모리가 1/4로 줄어듭니다. 그래서 큰 모델을 작은 GPU에 올리거나, 남는 메모리를 KV 캐시로 돌려 동시
  처리량을 높이는 데 유리해요.

  다른 양자화와 비교

  ┌─────────────┬───────┬─────────────────────────┬───────────────────────────────────────────────────┐
  │    기법     │ 비트  │          방식           │                       특징                        │
  ├─────────────┼───────┼─────────────────────────┼───────────────────────────────────────────────────┤
  │ AWQ         │ INT4  │ 활성값 기반 가중치만    │ 품질 좋음, 캘리브레이션 데이터 필요               │
  ├─────────────┼───────┼─────────────────────────┼───────────────────────────────────────────────────┤
  │ GPTQ        │ INT4  │ 2차 오차 기반 재구성    │ AWQ와 비슷, 역시 널리 쓰임                        │
  ├─────────────┼───────┼─────────────────────────┼───────────────────────────────────────────────────┤
  │ FP8         │ 8비트 │ 하드웨어 부동소수점     │ 품질 최고 수준, 단 메모리 절감은 절반만           │
  │             │       │                         │ (Ada/Hopper 필요)                                 │
  ├─────────────┼───────┼─────────────────────────┼───────────────────────────────────────────────────┤
  │ SmoothQuant │ INT8  │ 활성값↔가중치 스케일    │ 가중치+활성값 둘 다 양자화                        │
  │             │       │ 이동                    │                                                   │
  └─────────────┴───────┴─────────────────────────┴───────────────────────────────────────────────────┘

  비트 수 vs 정밀도:
  - AWQ/GPTQ(4비트) = 메모리 최대 절감, 품질 약간 손해
  - FP8(8비트) = 메모리 절반만, 품질 거의 손실 없음 + 하드웨어 가속

  아까 32B 얘기로 돌아가면

  L40S 48GB에서:

  ┌─────┬─────────────┬──────────────┬─────────────────────┐
  │     │ 가중치 크기 │ KV 캐시 여유 │     decode 속도     │
  ├─────┼─────────────┼──────────────┼─────────────────────┤
  │ FP8 │ ~33GB       │ ~8GB (좁음)  │ 빠름 (L40S HW 가속) │
  ├─────┼─────────────┼──────────────┼─────────────────────┤
  │ AWQ │ ~18GB       │ ~23GB (넉넉) │ 상대적으로 느림*    │
  ├─────────────┼───────┼──────────────────────────┼───────────────────────────────────────────────────────┤
  │ SmoothQuant │ INT8  │ 활성값↔가중치 스케일     │ 가중치+활성값 둘 다 양자화                            │
  │             │       │ 이동                     │                                                       │
  └─────────────┴───────┴──────────────────────────┴───────────────────────────────────────────────────────┘

  비트 수 vs 정밀도:
  - AWQ/GPTQ(4비트) = 메모리 최대 절감, 품질 약간 손해
  - FP8(8비트) = 메모리 절반만, 품질 거의 손실 없음 + 하드웨어 가속

  아까 32B 얘기로 돌아가면

  L40S 48GB에서:

  ┌─────┬─────────────┬──────────────┬─────────────────────┐
  │     │ 가중치 크기 │ KV 캐시 여유 │     decode 속도     │
  ├─────┼─────────────┼──────────────┼─────────────────────┤
  │ FP8 │ ~33GB       │ ~8GB (좁음)  │ 빠름 (L40S HW 가속) │
  ├─────┼─────────────┼──────────────┼─────────────────────┤
  │ AWQ │ ~18GB       │ ~23GB (넉넉) │ 상대적으로 느림*    │
  └─────┴─────────────┴──────────────┴─────────────────────┘

  * INT4는 연산 시 다시 FP16으로 dequantize해야 해서, L40S의 FP8 네이티브 가속만큼 빠르진 않음.

  정리:
  - 동시 처리량이 중요 → AWQ (KV 캐시 여유가 훨씬 많음 → 동시 요청 수↑)
  - 단일/소수 요청 속도가 중요 → FP8 (하드웨어 가속으로 토큰 생성 빠름)

  vLLM에서 쓰는 법

  # 사전 양자화된 AWQ 체크포인트 사용 (권장)
  vllm serve Qwen/Qwen2.5-32B-Instruct-AWQ \
    --quantization awq_marlin \
    --gpu-memory-utilization 0.92 \
    --max-model-len 16384 \
    --max-num-seqs 32 \
    --enable-prefix-caching

  - awq_marlin — vLLM의 최적화된 AWQ 커널. 그냥 awq보다 빠릅니다. 최신 vLLM은 AWQ 체크포인트를 자동으로 marlin 커널로
  로딩해요.
  - Qwen/Qwen2.5-32B-Instruct-AWQ — Qwen이 배포한 사전 양자화 버전. 직접 양자화할 필요 없음.
  - AWQ는 가중치가 작아 --max-num-seqs를 FP8(16)보다 높게(32~) 잡을 수 있습니다.

INT4 하드웨어 가속 지원하는 gpu 는?

⏺ NVIDIA Tensor Core INT4 지원 세대

  ┌───────────────────────┬───────────────────────────────┬──────────────────┬─────────────────────────────────────┐
  │       아키텍처        │           예시 GPU            │ INT4 Tensor Core │                비고                 │
  ├───────────────────────┼───────────────────────────────┼──────────────────┼─────────────────────────────────────┤
  │ Turing (SM75)         │ T4, RTX 2080, Quadro RTX      │        ✅        │ INT4/INT1 최초 도입                 │
  ├───────────────────────┼───────────────────────────────┼──────────────────┼─────────────────────────────────────┤
  │ Ampere (SM80/86)      │ A100, A30, A40, A10, RTX 3090 │        ✅        │ INT4/INT8/INT1                      │
  ├───────────────────────┼───────────────────────────────┼──────────────────┼─────────────────────────────────────┤
  │ Ada Lovelace (SM89)   │ L40S, L4, RTX 4090            │        ✅        │ INT4 + FP8 추가                     │
  ├───────────────────────┼───────────────────────────────┼──────────────────┼─────────────────────────────────────┤
  │ Hopper (SM90)         │ H100, H200                    │        ⚠️        │ FP8 중심, INT4는 사실상 미강조/축소 │
  ├───────────────────────┼───────────────────────────────┼──────────────────┼─────────────────────────────────────┤
  │ Blackwell (SM100/120) │ B200, GB200, RTX 5090         │      → FP4       │ INT4 대신 NVFP4/FP4로 진화          │
  └───────────────────────┴───────────────────────────────┴──────────────────┴─────────────────────────────────────┘

  → 당신의 L40S(Ada)는 INT4 Tensor Core를 지원합니다.

  ⚠️ 그런데 여기가 핵심 함정

  "INT4 Tensor Core 지원"과 "AWQ가 INT4 연산으로 빨라진다"는 다른 얘기입니다.

  LLM의 AWQ/GPTQ는 대부분 W4A16 방식입니다:
  - 가중치(Weight): INT4로 저장
  - 활성값(Activation): FP16 유지

  즉 실제 행렬곱은 INT4×INT4가 아니라, INT4 가중치를 FP16으로 되돌린(dequantize) 뒤 FP16 Tensor Core로 계산합니다. vLLM의
  Marlin 커널이 하는 게 정확히 이것이에요.

  그래서 AWQ의 속도 이점은:
  - ✅ 메모리 대역폭 절감 — 4비트라 가중치 로딩이 빠름 (decode가 메모리 병목이라 여기서 이득)
  - ❌ INT4 Tensor Core 연산 가속은 실제로 안 씀 (dequant 오버헤드가 오히려 붙음)

  순수 INT4×INT4 matmul(W4A4) 은 정확도가 너무 떨어져서 LLM에선 거의 안 씁니다. 그래서 INT4 Tensor Core 하드웨어가 있어도
  LLM 추론에선 그 유닛을 직접 활용하지 않는 게 일반적이에요.

  아까 L40S FP8 vs AWQ 비교로 연결

  이 때문에 앞서 말한 트레이드오프가 성립합니다:

  ┌────────────┬────────────────────────────────┬───────────────────────────────┐
  │            │           연산 방식            │        속도 이점 출처         │
  ├────────────┼────────────────────────────────┼───────────────────────────────┤
  │ FP8 (L40S) │ FP8 Tensor Core 네이티브 연산  │ 실제 연산 가속 → decode 빠름  │
  ├────────────┼────────────────────────────────┼───────────────────────────────┤
  │ AWQ (INT4) │ INT4→FP16 dequant 후 FP16 연산 │ 메모리 절감뿐, 연산 가속 없음 │
  └────────────┴────────────────────────────────┴───────────────────────────────┘

  → L40S에서 FP8이 decode 속도가 더 빠른 이유가 이거예요. FP8은 하드웨어가 그 정밀도로 직접 곱셈을 하지만, AWQ는 결국
  FP16으로 풀어서 계산하니까요.

  ---
  한 줄 요약: INT4 Tensor Core는 Turing·Ampere·Ada가 지원하고(Hopper는 FP8 중심, Blackwell은 FP4로 전환), L40S도 지원합니다.
  하지만 LLM의 AWQ는 INT4 연산이 아니라 W4A16(dequant→FP16)이라, INT4 Tensor Core를 실제로 쓰진 않고 메모리 절감 효과만
  봅니다.

  ⚠️ Hopper의 INT4 Tensor Core 지원 여부는 세대별 whitepaper마다 서술이 조금씩 다르니, 정확히 필요하면 최신 아키텍처 문서
  기준으로 확인해드릴 수 있습니다.
