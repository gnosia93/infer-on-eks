EKS 클러스터에서 초고속 네트워크인 EFA(Elastic Fabric Adapter) 트래픽을 모니터링하고 그라파나(Grafana)와 연동하는 것은 분산 학습이나 LLM 분산 추론(Ray, LWS 등)의 병목을 잡기 위해 매우 필수적인 작업입니다.
일반적인 네트워크 카드(ENA)와 달리, EFA는 커널을 우회하여 통신하므로 일반 Node Exporter로는 트래픽이 잡히지 않습니다. 따라서 AWS가 제공하는 전용 수집기를 배포해야 합니다. 전체적인 구현 아키텍처와 흐름을 정리해 드립니다.

![](https://github.com/gnosia93/infer-on-eks/blob/main/lesson/images/efa-monitoring-arch-1.png)

* EFA Node Exporter 배포 (EKS DaemonSet): 각 GPU 노드 내부의 /sys/class/infiniband 경로에서 EFA 상세 카운터 메트릭을 수집하여 포트로 노출합니다.
* Prometheus 가 스크랩(Scrape): 노출된 메트릭을 Prometheus(또는 Amazon Managed Prometheus)가 가져가 저장합니다.
* Grafana 시각화: 그라파나 대시보드에서 쿼리하여 모니터링합니다.

### 1단계: EKS 클러스터에 EFA용 Exporter 배포 ###
AWS 샘플 진영에서 관리하는 **efa-node-exporter**를 EKS 클러스터에 데몬셋(DaemonSet) 형태로 배포해야 합니다. 이 엑스포터는 GPU 노드마다 하나씩 뜨며 EFA 카드 상태를 긁어옵니다.
* GitHub 공식 소스: aws-samples/awsome-distributed-training 내의 efa-node-exporter 이용.
* 배포 방식: 쿠버네티스 DaemonSet 매니페스트를 작성하여 배포합니다. 호스트의 네트워크와 통신해야 하므로 hostNetwork: true 설정과 /sys 파일시스템 볼륨 마운트가 필요합니다.

```
# efa-node-exporter-daemonset.yaml (간략화된 예시)
apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: efa-node-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: efa-node-exporter
  template:
    metadata:
      labels:
        app: efa-node-exporter
    spec:
      hostNetwork: true
      hostPID: true
      containers:
      - name: efa-exporter
        image: public.ecr.aws/hpc-cloud/efa-node-exporter:latest # AWS 공개 이미지
        securityContext:
          privileged: true
        ports:
        - containerPort: 9100 # 메트릭 노출 포트
          name: metrics
        volumeMounts:
        - name: sys
          mountPath: /sys
          readOnly: true
      volumes:
      - name: sys
        hostPath:
          path: /sys
```

### 2단계: Prometheus 수집 설정 (Scrape Config) ###
배포된 efa-node-exporter가 뱉어내는 메트릭을 Prometheus가 긁어갈 수 있도록 PodMonitor 또는 Prometheus 설정을 추가합니다.
*	기본적으로 node-exporter와 유사한 방식으로 9100번 포트(또는 지정 포트)의 /metrics 엔드포인트를 타겟으로 잡습니다.

### 3단계: 어떤 메트릭을 봐야 할까? (핵심 지표) ###
그라파나 보드를 구성할 때, 단순히 트래픽 총량 외에 AWS가 최근 추가한 고도화된 EFA 손실 지표들을 꼭 모니터링해야 분산 추론 시 병목을 잡을 수 있습니다.
*	rdma_read_bytes / rdma_write_bytes: EFA를 통해 나아가고 들어온 실제 RDMA 데이터 처리량(Throughput).
*	tx_bytes / rx_bytes: 전체 전송/수신 바이트.
* retransmit_bytes / retransmit_pkts: 재전송된 패킷/바이트. 이 수치가 튀면 네트워크에 정체(Congestion)가 발생했거나 패킷 드랍이 일어나고 있다는 뜻입니다.
*	rx_dropped: 수신 버퍼 부족 등으로 드랍된 패킷 수.

### 4단계: Grafana 대시보드 연동 ###
그라파나 랩(Grafana Labs) 공식 홈페이지에 공식 또는 AWS 오피셜 샘플 대시보드가 등록되어 있습니다.
*	Grafana Dashboard ID: 20579 또는 23514 (EFA Metrics EKS)
*	방법: 1. 그라파나 웹 콘솔 접속
2. Dashboards ‭->  New ‭-> Import 선택
3. Import via grafana.com 창에 23514 입력 후 Load
4. 데이터 소스를 현재 EKS와 연동된 Prometheus로 지정하면 수동 제작할 필요 없이 즉시 화려한 EFA 트래픽 그래프가 완성됩니다.

### PromQL ###
EFA의 대표적인 가속 메트릭 카운터인 node_net_efa_tx_bytes, node_net_efa_rx_bytes, 그리고 드롭/에러 카운터를 활용합니다.📈 대역폭 (Bandwidth) 추이 쿼리각 노드의 EFA 인터페이스별 초당 송수신 처리량을 Gbps 단위로 변환하는 쿼리입니다.

* EFA 초당 송신(TX) 대역폭 (Gbps 단위):
```
sum(rate(node_net_efa_tx_bytes[1m])) by (instance, device) * 8 / 1024 / 1024 / 1024
```

* EFA 초당 수신(RX) 대역폭 (Gbps 단위):
```
sum(rate(node_net_efa_rx_bytes[1m])) by (instance, device) * 8 / 1024 / 1024 / 1024
```
* 🚨 패킷 드롭 및 에러 (Packet Drops & Errors) 쿼리
```
sum(rate(node_net_efa_tx_dropped[1m])) by (instance, device)
```

* 인터페이스별 초당 수신 패킷 에러 수:
```
sum(rate(node_net_efa_rx_errors[1m])) by (instance, device)
```

* Rate Window 조정: 위 쿼리는 [1m](1분 평균)을 기준으로 샘플링하고 있습니다. 만약 LLM 분산 배치 학습(All-Reduce 동기화 등) 중 발생하는 극단적인 초 단위 스파이크 트래픽을 관측하려면 Prometheus의 Scrape 주기를 5초~10초로 단축하고 범위를 [10s] 또는 [30s] 수준으로 좁혀서 설정하시는 것을 권장합니다.

* 🚨 Prometheus Alerting Rule 설정 (YAML)   
가장 중요한 알람 채널인 Slack(또는 PagerDuty, Webhook 등)과 연동할 수 있는 Prometheus Alerting Rule(경고 규칙) 구성 파일(YAML)입니다.LLM 분산 학습 환경에서 EFA의 패킷 드롭이나 에러 카운트 증가 현상은 학습 속도를 수십 배 저하시키거나 중간에 중단(Hang)시키는 주범이므로, 경고 등급을 나누어 관리하는 것이 핵심입니다.
```
apiVersion: ://coreos.com
kind: PrometheusRule
metadata:
  name: efa-monitoring-alerts
  namespace: monitoring
  labels:
    role: alert-rules
spec:
  groups:
  - name: efa.rules
    rules:
    
    # 1. WARNING: EFA 패킷 드롭 발생 (네트워크 혼잡 또는 일시적 병목 감지)
    - alert: EFAPacketDropWarning
      expr: sum(rate(node_net_efa_tx_dropped[1m])) by (instance, device) > 0.1
      for: 2m
      labels:
        severity: warning
      annotations:
        summary: "EFA 인터페이스에서 패킷 드롭이 감지되었습니다."
        description: "인스턴스 {{ $labels.instance }}의 EFA 장치 {{ $labels.device }}에서 최근 2분 동안 초당 {{ $value | printf \"%.2f\" }}개의 송신 패킷 드롭이 발생했습니다. 네트워크 충돌이나 대역폭 포화 상태를 확인하세요."

    # 2. CRITICAL: EFA 수신 에러 지속 (물리 인터페이스 불량 또는 심각한 패킷 유실)
    - alert: EFAReceiveErrorCritical
      expr: sum(rate(node_net_efa_rx_errors[1m])) by (instance, device) > 0
      for: 1m
      labels:
        severity: critical
      annotations:
        summary: "CRITICAL: EFA 수신 에러가 지속적으로 발생하고 있습니다!"
        description: "인스턴스 {{ $labels.instance }}의 EFA 장치 {{ $labels.device }}에서 하드웨어 수신 에러(초당 {{ $value }}건)가 감지되었습니다. 분산 학습 통신 오류 및 성능 저하가 발생 중일 수 있으니 즉시 인프라를 점검하십시오."

    # 3. CRITICAL: EFA 처리량 비정상 급감 (학습 프로세스 Hang 또는 통신 단절 의심)
    - alert: EFABandwidthDropDuringTraining
      expr: sum(rate(node_net_efa_tx_bytes[5m])) by (instance) == 0
      for: 5m
      labels:
        severity: critical
      annotations:
        summary: "CRITICAL: GPU 노드의 EFA 트래픽이 완전히 멈췄습니다."
        description: "GPU 인스턴스 {{ $labels.instance }}에서 실행 중인 워크로드의 EFA 데이터 송신량이 최근 5분간 0MB/s를 기록했습니다. 분산 학습 컨테이너의 데드락(Deadlock) 또는 네트워크 차단 여부를 파악해야 합니다."
```
* Slack 알림 메시지 포맷팅 가이드 (Alertmanager 설정용)
위에서 정의한 Alert 룰이 발동되면, 수신부인 Alertmanager가 이를 가공하여 Slack 채널로 쏘아줍니다. Slack 메시지를 읽기 쉽게 만들기 위한 Alertmanager의 alertmanager.yml 내부 템플릿 예시입니다.
```
receivers:
- name: 'slack-notifications'
  slack_configs:
  - channel: '#infra-alerts-efa'
    api_url: 'https://slack.com' # 본인의 Webhook URL로 대체
    send_resolved: true
    title: '[{{ .Status | toUpper }}] EFA 모니터링 알림 ({{ .CommonLabels.severity | toUpper }})'
    text: >-
      {{ range .Alerts }}
        *알림 내용:* {{ .Annotations.summary }}
        *상세 내용:* {{ .Annotations.description }}
        *발생 시간:* {{ .StartsAt.Format "2006-01-02 15:04:05" }}
      {{ end }}
```
